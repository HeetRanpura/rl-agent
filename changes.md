🧠 SECTION 2 — RL ENVIRONMENT & OPENENV DESIGN
Score: 6.0 / 10
The environment has a clear conceptual foundation — a real-world multi-step decision workflow with noise, boundary conditions, and escalation logic. The randomized persona system is a genuine strength. However, there are several design flaws that range from annoying to disqualifying.
Top 7 Design Flaws:
1. Task 5 self_reported_age is hardcoded to 35 despite claims of randomization. The code comment says "FIX D2: Age conflict is now randomised, not hardcoded" but the very next line is self_reported_age = 35 # always at PMKVY upper boundary. Only the Aadhaar age varies (36, 37, or 38). This means every Task 5 episode has the same surface profile (age=35), making memorization trivially easy. An agent that sees age=35 in a "DOCUMENT CONFLICT" task can always hard-code "request aadhaar_card → reject" and score perfectly. The claim of randomization is false.
2. The asyncio.Lock is defined but never acquired. Line 369 defines _state_lock = asyncio.Lock() but no code path ever does async with SchemeEnvEnvironment._state_lock. The lock is decorative. This means concurrent HTTP requests to /step or /reset will silently corrupt the shared class-level _shared_state dict. This is a latent bug that may not trigger in sequential single-agent evaluation but will if any test harness sends parallel requests.
3. Wrong escalation in Tasks 1–3 is non-terminal in the code. When an agent escalates incorrectly in Tasks 1–3, the environment returns done=False and applies a -2.0 reward, but the episode continues. This means a stubborn agent can "try" escalating, get penalized, and then continue — exploiting the non-terminal nature by farming partial rewards before eventually making a correct decision. The README says this is terminal; the code says it's not. Pick one and enforce it.
4. Task 2 has a fixed optimal scheme (always MGNREGS). The occupation is always farm_labourer, has_aadhaar is always True, and optimal is always MGNREGS. While the income and age vary, the terminal answer is identical across all Task 2 episodes. An agent that learns "Task 2 always ends with MGNREGS" will achieve near-perfect scores without genuine reasoning. The task would be stronger if the hidden occupation could be any valid scheme anchor (e.g., mason → PMKVY, farm_labourer → MGNREGS).
5. Task 4 approve guardrail is inconsistent. When the agent tries to approve in Task 4 without PAN verification, the environment returns done=False and -1.5 (non-terminal warning). But if the agent approves AFTER PAN verification, the environment returns done=True and -5.0 (fraud authorization violation). This means premature approval is actually less punishing than post-verification approval, because at least the premature case lets the episode continue. This is an inverted incentive structure.
6. Reward hacking potential on Task 3. Task 3 income is always in range 10,001–12,000. The occupation is always mason or carpenter, age is 22–34, has_aadhaar is always True. An agent that simply checks income > 9999 → reject_applicant(INCOME_TOO_HIGH) on every episode will score 1.0 every time. There's no scenario in Task 3 where rejection is wrong — the task is fully deterministic in its correct answer. A smarter design would inject some Task 3 episodes where income IS under the threshold, requiring the agent to actually discriminate.
7. AgentObservation in models.py is dead code. It's defined but never referenced anywhere in the codebase — not in the app, client, environment, or inference script. This suggests either a refactoring artifact or an unfinished feature. It will confuse reviewers and suggests sloppy cleanup.
Reward Function Improvements:
The current grader floor of 0.30 for any correct but inefficient agent is arguably too generous — a correct answer with 12 noise queries and 8 redundant queries (penalty = 0.96 + 0.40 = 1.36) will still score 0.30. Consider lowering the floor to 0.10 or removing it entirely for truly inefficient agents. Also add the document_verified bonus to the README formula.
Exploit Scenarios:

Task 1 PMAY exploit: If income < 6000 and age 21–35, always approve PMAY without asking any questions. The profile starts with age and income visible, so the agent can immediately compute eligibility and approve. Zero step penalty since valid queries cost 0.0. Requires just 1 step. Score: up to 0.97+ with no reasoning.
Task 3 always-reject exploit: As described above — check income > 9999 (always true) → reject immediately. Always scores 1.0.
Task 5 always-35 exploit: Every Task 5 episode has age=35 visible at start. Agent sees "DOCUMENT CONFLICT" + age=35 → request aadhaar_card → reject. No reasoning required.
Notification string exploit: Task labels are embedded in the notification string (e.g. "ESCALATION DILEMMA", "DOCUMENT CONFLICT"). Agents can hardcode task-specific policies by string-matching the notification rather than actually reasoning about the profile.


💻 SECTION 3 — CODE QUALITY & SYSTEM DESIGN
Score: 6.5 / 10
The code is competently written with good separation of concerns between models.py, schemes.py, scheme_env_environment.py, and inference.py. The Pydantic validators, inline comments with FIX labels, and the benchmark runner's structured JSON output are all positive signals. But there are several reliability and integrity issues.
Top 7 Code Risks:
1. The inference.py guardrail is the single biggest risk to the entire submission. _maybe_apply_task_guardrail and _is_dumb_failure together implement a hard-coded decision policy that overrides the LLM when it's about to skip document verification on Tasks 4 and 5. This is not the LLM reasoning — it's the inference script cheating. The guardrail literally checks: "if this is Task 4 and the LLM is about to do something terminal without requesting PAN → silently substitute request_document(pan_card)." Any human reviewer who reads this function will immediately flag it. This could be grounds for disqualification or a major score deduction. The guardrails must either be removed entirely or disclosed as a "scaffolding baseline" rather than presented as genuine agent behavior.
2. Import path inconsistency between app.py and the actual file layout. app.py imports from server.scheme_env_environment import SchemeEnvEnvironment and from server.schemes import SCHEMES, but scheme_env_environment.py and schemes.py are both present at the root level in the uploaded files. Meanwhile scheme_env_environment.py itself does from .schemes import SCHEMES (relative import) and from models import Action, Observation (absolute, relying on PYTHONPATH). The Dockerfile sets PYTHONPATH="/app/env:$PYTHONPATH" which would resolve the absolute imports, but only if the server/ package directory exists with __init__.py. The uploaded __init__.py content just exports SchemeEnvEnvironment, suggesting the package exists, but the file layout is ambiguous. If the server/ directory structure is wrong, the entire app will fail to start.
3. SCHEMES is imported in app.py but never used. This is dead import — it creates a runtime import dependency that adds startup time and signals poor code hygiene to reviewers.
4. _finalize_step comment contradicts the code. The comment says "FIX D5: changed >= to > to fix off-by-one" but the actual code uses >=. Either the fix was reverted without updating the comment, or the comment documents an intended change that wasn't applied. Either way, it's a reliability signal for reviewers.
5. The benchmark_runner.py step extraction has a dead loop. Lines 135–138 iterate over regex matches for step count extraction but the loop body is just pass with a comment "step count extraction handled in per-run analysis." This is cargo-cult code — the loop runs, does nothing, and the step extraction actually happens in extract_steps(). It should be deleted.
6. The _parse_action_response regex is fragile. The pattern r'\{[^{}]*"action_type"[^{}]*\}' will fail on any model that wraps a JSON object inside another JSON object (e.g., a chain-of-thought response that contains nested braces). The [^{}] character class means any curly brace invalidates the match. More robust parsing would use json.loads on multiple extracted candidates, or re.DOTALL with a broader match.
7. No tests exist despite pytest being in dev dependencies. The pyproject.toml lists pytest>=8.0.0 and pytest-cov>=4.0.0 as dev dependencies, and there's a project entry point defined, but zero test files are provided. For a competition submission claiming deterministic grading and reproducible behavior, the absence of even basic unit tests for get_eligible_schemes(), _compute_grader_score(), or the environment's step logic is a credibility issue.
Refactoring Suggestions:
Remove or clearly label the guardrail functions in inference.py. Fix app.py to remove the unused SCHEMES import. Delete the dead loop in benchmark_runner.py. Add a tests/ directory with at minimum: test_scheme_eligibility.py testing boundary conditions (income=9999, income=10000, age=35, age=36), and test_grader_score.py testing score computation. Fix the asyncio.Lock to actually be used in step() and reset(), or remove it to avoid false safety signals.
Critical Bugs / Failure Points:
If the server/ package directory doesn't exist exactly as the import paths expect, the Docker container will fail with ModuleNotFoundError at startup — a silent deployment failure. The asyncio.Lock non-use means concurrent evaluation requests will corrupt episode state. The guardrail override will be caught by any reviewer who runs inference and logs what actions are actually being sent to the environment vs. what the LLM suggested.

Section 2 — RL Environment Design (7.5/10)
OpenEnv Compliance
The OpenEnv integration is solid. The step(), reset(), and state property are correctly implemented. The Observation and Action models are properly Pydantic-typed with validators. The openenv.yaml is complete. The _post-style REST interface in inference.py correctly mirrors the expected /reset and /step endpoints.
One compliance issue: scheme_env_environment.py uses from models import Action, Observation without a package prefix, which works only when Python is run from /app/env with PYTHONPATH=/app/env. The __init__.py properly exports SchemeEnvEnvironment, but a developer running python server/app.py from the repo root hits a ModuleNotFoundError. The correct fix is from scheme_env.models import ... throughout the server package, consistent with how pyproject.toml defines the package layout.
Environment Design Quality
The partial observability model is thoughtfully designed. The missing_data list as a first-class observation field is a good design choice — the agent always knows what it doesn't know. The noise field injection (1–3 random irrelevant fields in known_profile) is a genuine test of attention, not just a cosmetic addition.
The four-field action space (ask_question, request_document, approve_scheme, reject_applicant, escalate) maps cleanly to the real-world workflow and is neither too large nor too small. The Pydantic validator in Action that enforces allowed values per action type is good defensive design.
However, the single-episode evaluation is a serious reliability problem. Each task is evaluated exactly once per inference run. Because generate_dynamic_persona randomizes age, income, occupation, and employer each call, two evaluations of the same model on the same task can differ by ~0.15 in grader_score purely due to persona luck — not model capability. A model that happens to get a Task 3 persona with income=10001 (just barely over threshold, obvious signal) vs income=11999 (same logical answer, noisier profile) may score differently. The benchmark should run each task 3–5 times and report mean ± std, especially since the benchmark_runner already loops over multiple models. The code infrastructure is all there; it just needs an outer repetition loop.
The Task 5 persona has a frozen self_reported_age = 35 despite a comment that it was fixed: the comment says "FIX D2: Age conflict is now randomised" but the code is self_reported_age = 35  # always at PMKVY upper boundary. This is a half-fix. The Aadhaar age is randomized (36, 37, or 38), but the self-reported age is always 35. A model that has been evaluated even once on this task could infer the pattern.
Reward Engineering
The reward structure shows good thinking. The -0.10 per noise query, combined with the efficiency penalty in _compute_grader_score, creates a meaningful gradient against lazy information gathering. The base_score - penalty formula with a max(0.30, ...) floor prevents catastrophic scores for correct-but-inefficient agents, which is exactly the right policy for a benchmark meant to reward both correctness and efficiency.
The non-terminal soft-block mechanism (Task 4: reject before PAN → reward=-1.0, done=False) is a sophisticated design. It teaches the agent a protocol without terminating the episode on the first mistake. This is clearly the product of iteration.
Exploit scenario 1 — Task 3 always-reject: An agent that learns "always reject on Task 3" without ever reading income gets a non-zero score from the environment because the check is if "income" not in obs.known_profile: score=0.0 at line 722 — but income IS in known_profile for Task 3 (the persona is fully provided). So the income field is already known from reset(). The check passes trivially. A pathological agent that skips all reasoning and directly calls reject_applicant("INCOME_TOO_HIGH") on any task with income visible in the profile would score 1.0 on Task 3 with zero information gathering. The benchmark_runner flags this as POSSIBLE_T3_EXPLOIT which is good awareness, but the environment itself doesn't penalize it.
Exploit scenario 2 — Task 4 guardrail bypass in inference.py: The most serious issue in the entire project. _maybe_apply_task_guardrail and _is_dumb_failure in inference.py silently override the model's action when it tries to make a terminal decision on Tasks 4/5 before the required document has been verified. Specifically, if the model tries to escalate on Task 4 without having requested the PAN card, the inference script intercepts the action and substitutes request_document("pan_card") automatically. This means the model never actually learns to do this — the script does it for the model and reports the resulting score as the model's performance. This is evaluation pollution. Task 4 and Task 5 scores as reported by this runner are not honest capability measurements; they partially reflect the guardrail's intervention. A hackathon evaluator who inspects the inference logs and sees that the request_document action was generated by the guardrail rather than the model will flag this immediately.
Task & Grader System
The grader design using _compute_grader_score with noise_queries × 0.08 + redundant_queries × 0.05 + wasted_steps × 0.04 is clean and interpretable. The max(0.30, ...) floor is a reasonable policy. However, because random.randint is seeded only by task_id (via env_reset(task) which sends {"seed": task}) and the actual random calls in generate_dynamic_persona are unseeded, the same integer task seed produces different personas on each call. This is probably intentional for generalization, but it means grader scores are stochastic across runs.

Section 3 — Code Quality & Architecture (6.5/10)
Structure and Modularity
The project has reasonable modularity: schemes.py (eligibility rules), models.py (Pydantic types), scheme_env_environment.py (environment logic), app.py (FastAPI server), client.py (EnvClient), inference.py (agent loop), benchmark_runner.py (multi-model orchestration). The separation of get_eligible_schemes() and get_optimal_scheme() into a standalone schemes.py is good — it means eligibility logic can be unit tested independently.
The _finalize_step method as a single exit point for all step branches is good defensive design that ensures step count enforcement and state persistence on every path.
Critical Bugs and Failure Points
Import path inconsistency: scheme_env_environment.py line 9 does from .schemes import SCHEMES (relative import) but line 10 does from models import Action, Observation (absolute, package-less). In the Docker container with PYTHONPATH=/app/env, the absolute import resolves — but it is fragile and will fail in any other execution context. The relative import pattern should be consistent throughout: from .models import Action, Observation.
The history list in inference.py is never reset between tasks. In main(), run_episode(task) is called for tasks 1, 2, 3, 4, 5 sequentially. Each call creates a fresh history = [] inside run_episode, which is correct. However, get_agent_action passes history[-10:] to the LLM. This is fine per-episode, but if run_episode ever throws and is caught by the outer try/except in main, the history state is lost but the environment might still have an open episode. This is low-risk but worth noting.
extract_steps in benchmark_runner.py uses task index counting (incrementing task_idx on every [END] line) rather than parsing the task name from the log. If any task crashes and logs two [END] lines or zero, the counting drifts and step counts get assigned to the wrong tasks. This is a silent data corruption bug in the report.
The asyncio import in benchmark_runner.py is used only for asyncio.create_subprocess_exec and asyncio.sleep — the runner is effectively sequential (MAX_CONCURRENT = 1, the for loop awaits each model before the next). The async scaffolding adds complexity without benefit. A simple subprocess.run loop would be cleaner and easier to debug.
benchmark_runner.py hardcodes the model list (MODELS_TO_TEST = [...]). This means changing which models to test requires editing the source file. A simple --models CLI flag or an environment variable would make this much more usable.
No unit tests. The entire eligibility logic in get_eligible_schemes is deterministic and pure — it's trivially testable. Edge cases like income=9999 (should qualify for PMKVY), income=10000 (should not), age=35 (should qualify), age=36 (should not) are exactly the boundary conditions that need automated tests. The pytest infrastructure is in pyproject.toml but zero test files exist.
Inference Script Quality
The SYSTEM_PROMPT is well-written and specific. The instruction to respond with exactly one JSON object with no fences is good. Embedding the eligibility rules directly in the system prompt (rather than relying on training data) is the right call for ensuring consistent behavior across model families.
The _parse_action_response regex r'\{[^{}]*"action_type"[^{}]*\}' picks the last match (via matches[-1]). For models that emit reasoning tokens before the JSON, this correctly discards the preamble. But it also means a model that emits two JSON objects (e.g. one for thought, one for action) will use the latter — which is usually correct but could backfire if the model writes a summary JSON after the action JSON.
The time.sleep(0.3) between steps is conservative but correct for HF Router rate limits.
Dockerfile and Deployment
The multi-stage Dockerfile is well-structured. Using uv sync --frozen for reproducibility is good practice. The HEALTHCHECK with curl -f http://localhost:7860/health is correctly implemented. The sha256-pinned base image is excellent for reproducibility. The ENABLE_WEB_INTERFACE=true ENV is a nice touch for HF Spaces integration.
One issue: the Dockerfile does two separate uv sync calls (lines 23–28 and 30–35) — once without the project and once with. This is likely intentional to exploit Docker layer caching (dependencies layer separate from project files), but the comments don't explain this, making it appear like a duplicate.

Final Meta-Evaluation (69/100)
Top 5 reasons this project could fail:

The inference guardrail in _maybe_apply_task_guardrail silently overrides model actions on Tasks 4 and 5. If the evaluation framework runs inference end-to-end and inspects logs, it will detect that the reported Task 4/5 scores are partially the guardrail's work, not the model's. This is the single most damaging issue.
Task 5 is not in the README. If the evaluation criteria checks "are all tasks documented?", this fails immediately. The README says "5 tasks" in the title but only defines 4. The openenv.yaml also doesn't mention a task count.
Zero baseline results. Without numbers, a human reviewer cannot assess whether the environment is calibrated correctly. A well-designed RL benchmark should show that random agents score near 0, weak models score 0.2–0.4, and strong models score 0.7+.
The import path inconsistency (from models import vs from .models import) means the server will silently fail to start in any execution context other than the Docker container with the exact PYTHONPATH set in the Dockerfile.
Single-episode evaluation produces noisy scores. Two runs of the same model on the same task can differ by 0.10–0.20 due to persona randomization. Without multi-episode averaging, the leaderboard numbers are not statistically meaningful.

Top 5 improvements that would 10x the project:

Remove the guardrail entirely from inference.py. Replace it with a note in the README: "Task 4 requires the agent to proactively request the PAN card without being prompted." Report honest Task 4/5 scores. This single change turns a questionable benchmark into an honest one.
Add Task 5 to the README with the same format as Tasks 1–4: objective, challenge, minimum steps, grader formula.
Run 5 repetitions per task per model and report mean ± std. The benchmark_runner already has the infrastructure; add an outer for repeat in range(N_REPEATS) loop and compute standard deviation in the aggregate report.
Add 10 unit tests for get_eligible_schemes and _compute_grader_score. Cover all income/age boundary conditions. This would take 2 hours and would immediately surface any future regression.
Publish actual baseline numbers from at least two models (a weak 7B and a strong 70B) as a table in the README. Ideally show task-level scores, not just average, so the difficulty progression is visible.
